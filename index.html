Recap: Version Control System

Commit: A change set.

Branch: A sequence of commits representing edit history.

Repository: A place (directory) where branches live.

Commit

Basically can be imagined as a diff file containing changes in multiple files.

Branch

A series commits that may branch out and be merge with the others.

Repository

A place where all branches live, which is named .git

What is Github?

  • It is online platform holding repositories.
  • It is a project management platform for
    • hosting documents
    • communicating bugs and features
    • profiling the work load
  • You can even setup static website on it.

Clone or Fork?

  • They both create a mirroring repository.
    • Branches (history)
    • Working directory (all the files being tracked)

Pull and Push

  • Both are used to synchronize the history (branch)
    • Push will update REMOTE repository with a local branch.
    • Pull will update LOCAL repository with a remote branch.

Basic flow

  • git init Create a repository with in the project root directory.
  • git add . Add changes to staging area, i.e., create a snapshot of the changes you have made.
  • git commit -m "your commit message" Commit the staged changes to repository.
  • git remote add <name> <URL> Add a reference to a remote repository.
  • git push <name> master Upload (synchronize) current branch to remote repository.

Note: You should have the upstream branch setup for current branch when using git push.

When something happened

  • git log Viewing the history of current branch.
  • git checkout <ref> Update the working directory to the states of a branch/tag/commit.
  • git revert <ref> Create a new commit to revert the changes made since ref.
  • git checkout -b <new branch> Create a new branch from current point.
  • git merge <branch> Merge the changes from another branch.

Creating a git repository

  • LOCAL: git init & git clone
  • GITHUB: Create and Fork

Prepare for commit

  • Staging the difference:
    git add <filepath>.
  • Create a commit in the repository.
    git commit -m "<description>"

Synchronizing reposiory

  • Create a local mirroring repository:
    git clone <URL> <dir>
  • Pull (download) from the upstream:
    git pull
  • Push (upload) to the upstream:
    git push

Forking on Github?

It is just git clone on Github.

Day to Day Routine

  • Always pull before push to make sure the changes in the remote branch integrates with your new changes.
    • Resolve conflicts.
    • Test integrity.
    • Keep remote master consistent all the time.
  • Open issues to document the discussions and decision making processes for later reference.
